  <action>
1. Create `internal/launch/reporter.go` (NO build tag -- shared interface):
   - Define `StepStatus` type (int) with constants: `StepPending`, `StepRunning`, `StepDone`, `StepFailed`, `StepSkipped`
   - Define `ProgressReporter` interface:
     ```go
     type ProgressReporter interface {
         StepStarted(name string)
         StepCompleted(name string)
         StepFailed(name string, err error)
         StepSkipped(name string)
         StepPaused(name string)
     }
     ```

2. Create `internal/launch/reporter_cli.go` (NO build tag):
   - Define `CLIReporter` struct with explicit fields:
     ```go
     type CLIReporter struct {
         activeSpinner *ui.StepSpinner
     }
     ```
   - Implement `ProgressReporter` interface:
     - `StepStarted(name)`: assigns `c.activeSpinner = ui.StartStep(name)` (creates and starts new spinner)
     - `StepCompleted(name)`: calls `c.activeSpinner.Success()`
     - `StepFailed(name, err)`: calls `c.activeSpinner.Fail()`
     - `StepSkipped(name)`: calls `c.activeSpinner.Success()` (same visual as completed)
     - `StepPaused(name)`: calls `c.activeSpinner.Stop()` (stops spinner for progress bar display, e.g., during download)
   - Constructor: `func NewCLIReporter() *CLIReporter { return &CLIReporter{} }`

3. Modify `internal/launch/pipeline.go`:
   - Add `RunWithReporter(ctx context.Context, cfg *config.Config, reporter ProgressReporter) error` function
   - This function contains the core pipeline logic currently in `Run()`, but uses the reporter instead of directly calling spinner functions
   - The existing `Run()` function becomes a thin wrapper: creates a `NewCLIReporter()` and calls `RunWithReporter(ctx, cfg, reporter)`
   - In the step loop, replace `spinner := ui.StartStep(step.Name)` with `reporter.StepStarted(step.Name)`, and replace `spinner.Fail()`/`spinner.Success()` with `reporter.StepFailed()`/`reporter.StepCompleted()`
   - The `Step` struct's `Fn` signature needs adjustment: currently takes `*ui.StepSpinner` as parameter. Change to accept `ProgressReporter` instead, OR remove the spinner parameter entirely and have steps that need spinner access (like stepDownloadGame which calls spinner.Stop()) use reporter methods instead. Best approach: remove the `*ui.StepSpinner` parameter from step functions.
   - Update `Step.Fn` signature to `func(ctx context.Context, state *LaunchState) error` (no spinner/reporter param -- steps that need to pause use the reporter stored on LaunchState)
   - Add `Reporter ProgressReporter` field to `LaunchState` so steps can access it
   - Update ALL step functions to remove the spinner parameter:
     - `stepHealthCheck`: no change needed (doesn't use spinner directly)
     - `stepAuthenticate`: was calling `spinner.Stop()` before prompts -- now call `state.Reporter.StepPaused("Authenticating")` or handle differently (in GUI mode, auth is pre-done so this path won't be hit)
     - `stepOIDCToken`: no change (doesn't use spinner)
     - `stepBootstrap`: no change (doesn't use spinner)
     - `stepCheckVersion`: no change
     - `stepDownloadGame`: was calling `spinner.Stop()` before progress bar -- now call `state.Reporter.StepPaused("Downloading game update")` or similar
     - `stepLaunchGame`: no change

IMPORTANT: This is a refactor of existing code. The existing CLI behavior MUST remain identical. The `Run()` function must produce the exact same terminal output as before. Only the internal wiring changes.

IMPORTANT: Also update platform-specific step functions in `pipeline_linux.go` and `pipeline_windows.go` to match the new `Step.Fn` signature (remove `*ui.StepSpinner` parameter).

     d. On failure: update error label text inside `fyne.Do()`, re-enable button

   - Wrap form elements in `container.NewVBox()` with `container.NewCenter()` for horizontal centering

   - Check `auth.LoadCredentials()` at startup

   - Handle the `onSuccess` callback to transition from login to main view via `w.SetContent()`

IMPORTANT: All widget updates from goroutines (login handler) MUST be wrapped in `fyne.Do()` to avoid data races (Fyne v2.6+ threading model).
