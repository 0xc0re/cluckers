# Phase 07.1 Plan 04: Hardware Validation — FAIL

**Controller buttons do NOT persist through ServerTravel on any tested approach. Root cause confirmed as Steam Input firmware reconfiguration — a firmware-level issue that software at the evdev/XInput layer cannot fix.**

## Performance

- **Duration:** ~6 hours across 5 debugging sessions
- **Started:** 2026-02-25
- **Completed:** 2026-02-27
- **Hardware deploys:** 14 (5 evdev proxy, 1 baseline, 5 XInput DLL, 3 clean)
- **Approaches tested:** 3 (evdev proxy, XInput DLL proxy, clean baseline)

## Result: FAIL — CTRL-03 NOT Satisfied

Controller buttons (A/B/X/Y, bumpers, triggers) lose input during UE3 ServerTravel (lobby-to-match transition) on Steam Deck. This occurs regardless of:
- evdev-to-uinput proxy with dead reckoning
- XInput DLL proxy with backend forwarding
- Clean Steam-managed Proton launch (no intervention at all)

## Hardware Deploy Log

### evdev Proxy (Deploys 1-8)
| Deploy | Config | Result |
|--------|--------|--------|
| 1-4 | Iterating on proxy code | Build/setup issues |
| 5 | Proxy creates uinput Xbox 360 pad | Steam Input virtual pads **destroyed** |
| 6 | Retry with timing changes | Same — uinput kills Steam Input |
| 7 | Different device parameters | Same fundamental issue |
| 8 | Final attempt | **Abandoned** — uinput gamepad + Steam Input incompatible |

**Root cause:** Creating ANY uinput gamepad device (VID=045e PID=028e) causes Steam Input to permanently tear down its own virtual pads (VID=28de PID=11ff). The proxy's output device kills the proxy's input source.

### Baseline (Deploy 9-10)
| Deploy | Config | Result |
|--------|--------|--------|
| 9 | Direct `proton run` (no Steam) | Controller works in menu, drops in match |
| 10 | Steam-managed, misspelled `WINEDLOVERRIDES` | Same as baseline — env var typo meant no overrides active |

### XInput DLL Proxy (Deploys 11-13)
| Deploy | Config | Result |
|--------|--------|--------|
| 11 | DLL loaded, index remap active | **Doubled input** — game polls indices 0+1, remap maps both to real 0 |
| 12 | DLL without DEF file | **Crash** — ordinal mismatch, "filename not found in xinput1_4.dll" |
| 13 | DLL with DEF file, no remap | Sticks work (LX=1193), **btns=0000 always**. Crash after 936 calls (~8s) |

**Root cause:** Our DLL loads `xinput1_4.dll` as backend, bypassing Proton's custom Steam Input IPC in builtin `xinput1_3.dll`. Proton's XInput DLLs have custom button translation via Steam Input; loading alternatives gets raw/partial data (sticks yes, buttons no).

### Clean Baseline (Deploys 14a-14b)
| Deploy | Config | Result |
|--------|--------|--------|
| 14a | No DLL, `WINEDLLOVERRIDES="dxgi=n"` | **Instant crash** — Proton manages DXVK internally |
| 14b | No DLL, no overrides, `cluckers prep && %command%` | Controller works in menu, **drops during ServerTravel** |

**Conclusion:** Even with zero intervention (pure Steam-managed Proton launch), controller drops during ServerTravel. This is a firmware-level issue.

## Three Fundamental Incompatibilities Discovered

1. **evdev/uinput + Steam Input:** Creating any uinput gamepad triggers Steam Input to tear down its virtual pads. The proxy output device destroys the proxy input source.

2. **XInput DLL proxy + Proton:** Proton's xinput DLLs contain custom Steam Input IPC for button translation. Loading `xinput1_4.dll` as a backend bypasses this, resulting in sticks-only data with zeroed buttons.

3. **WINEDLLOVERRIDES + Proton-managed launch:** Proton manages its own DLL overrides internally. External `dxgi=n` conflicts with Proton's DXVK management and causes instant crash.

## Additional Discovery

**GE-Proton10-32 already has XInputEnable(FALSE) no-op fix** (merged in Proton 8.0-4 for Overwatch 2). The custom XInput DLL's primary purpose — blocking XInputEnable(FALSE) calls — was always redundant under Proton-GE.

## Decisions Made

- **evdev proxy abandoned** after 4 deploys confirmed uinput kills Steam Input virtual pads
- **XInput DLL proxy abandoned** after discovering it bypasses Proton's Steam Input IPC
- **Index remapping removed** — unnecessary under Proton/Steam Input, causes doubled input
- **WINEDLLOVERRIDES removed entirely** — Proton manages DLLs internally
- **Controller persistence deferred to v1.2+** — firmware-level issue, workarounds: external USB controller, launch through Steam full runtime, or await Valve fix

## Phase 7.1 Outcome

Plans 01-03 produced working, well-tested proxy infrastructure (18 unit tests, clean pipeline integration). Plan 04 hardware validation proved the proxy cannot solve the underlying problem — Steam Input firmware reconfiguration during ServerTravel is triggered by Gamescope losing track of the game window, and no intervention at the evdev/XInput/DLL layer can prevent this.

The correct fix requires either:
- Launching through Steam's full game runtime (proper AppID → Gamescope window tracking)
- Valve fixing Steam Input firmware behavior for non-Steam games doing ServerTravel
- An external USB controller (bypasses Steam Input firmware entirely)
