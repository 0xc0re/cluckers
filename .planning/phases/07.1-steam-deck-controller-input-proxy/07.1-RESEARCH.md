# Phase 07.1: Steam Deck Controller Input Proxy - Research

**Researched:** 2026-02-25
**Domain:** Linux evdev/uinput virtual gamepad, Steam Input event forwarding, dead reckoning
**Confidence:** MEDIUM

## Summary

This phase addresses CTRL-03: controller buttons must persist through lobby-to-match transitions on Steam Deck. The root cause is well-understood from extensive prior debugging (see controller-debugging.md): Steam Input reconfigures the Steam Deck controller firmware during UE3 ServerTravel, zeroing button data (HID bytes 8-13) at the hardware level. Axes continue to work because they occupy different HID bytes (48-55).

The solution is a pure Go uinput proxy that sits between Steam Input's virtual Xbox 360 pad (evdev event7, VID 28de PID 11ff) and the game. The proxy creates a new virtual Xbox 360 gamepad via `/dev/uinput`, reads events from the Steam Input virtual pad, forwards them, and when it detects an all-zeros button state (the ServerTravel signature), holds the last known button state for a configurable timeout instead of forwarding zeros. Wine's winebus is configured via registry to only see the proxy's virtual pad (DisableHidraw=1, Enable SDL=1).

**Primary recommendation:** Implement a ~300 LOC `internal/launch/inputproxy` package using raw syscalls for uinput (kenshaw/evdev has incomplete uinput support) and kenshaw/evdev for evdev reading. Integrate as a pipeline step that starts before game launch and stops after game exit.

### Pattern 1: Evdev-to-Uinput Proxy with Dead Reckoning
**What:** A goroutine reads evdev events from the source device, maintains shadow state, and forwards events to a virtual uinput device. When all buttons simultaneously go to zero (the ServerTravel signature), the proxy holds the previous button state for a configurable timeout.
**When to use:** When the source device has transient data loss at the firmware level.
**Example:**
```go
// Source: Linux kernel uinput documentation + prior evdev.go from feature/deck-controller
type InputProxy struct {
    source    *evdev.Evdev     // Steam Input virtual pad (read)
    sink      *os.File         // /dev/uinput virtual gamepad (write)
    buttons   uint32           // Shadow button state
    lastBtn   time.Time        // When buttons last had non-zero value
    holdTime  time.Duration    // How long to hold state on zero detection
    stopCh    chan struct{}
    done      chan struct{}
}

func (p *InputProxy) Run(ctx context.Context) {
    for envelope := range p.source.Poll(ctx) {
        ev := envelope.Event
        switch ev.Type {
        case evdev.EventKey:
            // Button event -- update shadow state
            p.updateButton(ev.Code, ev.Value)
            if p.allButtonsZero() && time.Since(p.lastBtn) < p.holdTime {
                continue // Hold: don't forward the zero
            }
            p.forward(ev)
        case evdev.EventAbsolute:
            p.forward(ev) // Axes always forwarded (they work during ServerTravel)
        case evdev.EventSync:
            p.forwardSync()
        }
    }
}
```

### Pattern 3: Wine Registry Patching for Device Isolation
**What:** Write Wine registry entries to ensure the game only sees the proxy's virtual pad.
**When to use:** After Proton compatdata is created but before game launch.
**Example:**
```go
// Create a .reg file and import via `proton run regedit`
// OR directly write to compatdata/pfx/system.reg (text format)
//
// Registry key: HKLM\System\CurrentControlSet\Services\winebus
//   "DisableHidraw" = dword:00000001
//   "Enable SDL"    = dword:00000001

func patchWinebusRegistry(compatdataPath string) error {
    regFile := filepath.Join(compatdataPath, "pfx", "winebus_patch.reg")
    content := `Windows Registry Editor Version 5.00

[HKEY_LOCAL_MACHINE\System\CurrentControlSet\Services\winebus]
"DisableHidraw"=dword:00000001
"Enable SDL"=dword:00000001
`
    os.WriteFile(regFile, []byte(content), 0644)
    // Import via: WINEPREFIX=<pfx> wine regedit <regFile>
    // Or via: proton run regedit <regFile>
}
```

### Pattern 4: Device Detection by VID/PID
**What:** Scan /dev/input/event* devices and match by vendor/product ID to find the Steam Input virtual pad.
**When to use:** At proxy startup, before the game launches.
**Example:**
```go
// Source: prior findXboxPadDevice() from feature/deck-controller + kenshaw/evdev API
func findSteamInputPad() (string, error) {
    // Scan /proc/bus/input/devices for VID=28de PID=11ff (Steam Virtual Gamepad)
    // OR use kenshaw/evdev:
    for i := 0; i < 20; i++ {
        path := fmt.Sprintf("/dev/input/event%d", i)
        dev, err := evdev.OpenFile(path)
        if err != nil {
            continue
        }
        id := dev.ID()
        if id.Vendor == 0x28de && id.Product == 0x11ff {
            dev.Close()
            return path, nil
        }
        dev.Close()
    }
    return "", fmt.Errorf("Steam Input virtual pad not found")
}
```

### Anti-Patterns to Avoid
- **Reading from hidraw**: Steam holds exclusive access to hidraw2. Read from evdev event node instead.
- **Grabbing (EVIOCGRAB) the source device**: This would prevent other consumers. The proxy should passively read.
- **Using STEAM_INPUT_DISABLE=1**: This stops the virtual pad entirely. Steam Input must stay active.
- **Running winetricks in Proton prefix**: Never. Use regedit for registry changes.
- **Forwarding SYN_DROPPED events**: When kernel drops events, re-read full device state instead.
- **Trusting kenshaw/evdev for uinput creation**: Its WithAbsoluteTypes is a no-op. Use raw ioctls.

## Common Pitfalls

### Pitfall 1: Axis Y-Inversion
**What goes wrong:** Evdev Y-axis is inverted relative to XInput (evdev Y+ = down, XInput Y+ = up).
**Why it happens:** Linux evdev and Windows XInput use opposite Y-axis conventions for joysticks.
**How to avoid:** Negate Y values when forwarding ABS_Y and ABS_RY events (with -32768 -> 32767 clamping).
**Warning signs:** Camera or movement inverted vertically in-game.

### Pitfall 2: Trigger Range Mismatch
**What goes wrong:** Game expects 0-255 triggers but evdev reports 0-1023 for some devices.
**Why it happens:** Steam Input's virtual Xbox 360 pad may use different ranges than physical xpad.
**How to avoid:** Read actual axis ranges from source device via evdev AbsoluteTypes(), scale to 0-255 output range.
**Warning signs:** Triggers stuck at partial values or not reaching full pull.

### Pitfall 3: SYN_DROPPED Handling
**What goes wrong:** Kernel drops events when consumer is too slow, resulting in stale state.
**Why it happens:** evdev has a fixed-size ring buffer. If consumer falls behind, events are dropped.
**How to avoid:** kenshaw/evdev handles SYN_DROPPED internally. If using raw reads, re-read full device state via EVIOCGABS ioctls on drop.
**Warning signs:** Buttons appear stuck or unresponsive intermittently.

### Pitfall 4: Dead Reckoning False Positive
**What goes wrong:** Player releases all buttons legitimately and proxy holds phantom buttons.
**Why it happens:** Cannot distinguish "player released all buttons" from "Steam Input zeroed buttons."
**How to avoid:** Use a short hold timeout (200-500ms). ServerTravel transition is >1 second, so holding for 500ms covers the gap without noticeable phantom input. Also only trigger hold when ALL buttons AND triggers go to zero simultaneously (single button release is normal).
**Warning signs:** Ghost button presses after match transition, delayed button release in menus.

### Pitfall 5: Device Permission Denied
**What goes wrong:** Cannot open /dev/uinput or /dev/input/event* without root.
**Why it happens:** Missing udev rules or `input` group membership.
**How to avoid:** On SteamOS (Steam Deck), udev rules from the `steam-devices` package grant uaccess to /dev/uinput via rule: `KERNEL=="uinput", SUBSYSTEM=="misc", TAG+="uaccess", OPTIONS+="static_node=uinput"`. Input event nodes also have uaccess. On desktop Linux, user must be in `input` group.
**Warning signs:** "permission denied" errors at proxy startup. Proxy should fail gracefully (non-fatal) with instructions.

### Pitfall 6: Wine Sees Both Real and Virtual Pad
**What goes wrong:** Game receives duplicate/conflicting input from both the Steam Input pad and the proxy pad.
**Why it happens:** Wine's winebus enumerates all available input devices by default.
**How to avoid:** Set Wine registry entries: DisableHidraw=1 (prevents winebus from reading physical devices), Enable SDL=1 (winebus uses SDL which sees our uinput device). This is NOT the same as STEAM_INPUT_DISABLE=1.
**Warning signs:** Double input, buttons registering twice, input desync.

### Pitfall 7: Proxy Outlives Game Process
**What goes wrong:** Virtual gamepad device persists after game exit, confusing subsequent launches.
**Why it happens:** Proxy goroutine not properly cleaned up, or uinput fd not closed.
**How to avoid:** Use context cancellation tied to game process lifecycle. Close uinput fd in defer. The kernel automatically destroys the virtual device when fd is closed.
**Warning signs:** Multiple virtual gamepads accumulating across launches.

### Dead Reckoning State Machine
```go
// Source: Prior investigation documented in controller-debugging.md
// ServerTravel signature: ALL buttons + triggers go to zero simultaneously
// while axes continue (HID bytes 8-13 zero, bytes 48-55 non-zero)

type buttonState struct {
    buttons  uint32     // Bitmask of currently pressed buttons
    lTrig    uint8      // Left trigger value
    rTrig    uint8      // Right trigger value
    lastNonZero time.Time  // Last time any button/trigger was non-zero
}

func (s *buttonState) isServerTravelZero() bool {
    return s.buttons == 0 && s.lTrig == 0 && s.rTrig == 0
}

func (s *buttonState) shouldHold(timeout time.Duration) bool {
    // Only hold if we recently had non-zero button state
    // AND all buttons/triggers went to zero simultaneously
    return s.isServerTravelZero() && time.Since(s.lastNonZero) < timeout
}
```

## Open Questions

1. **Proton regedit invocation for registry patching**
   - What we know: `wine regedit` imports .reg files. Proton wraps wine as `proton run`.
   - What's unclear: Does `proton run regedit /S file.reg` work correctly? Does the registry persist across launches? Is there an alternative like writing directly to system.reg text file?
   - Recommendation: Test both approaches. Direct system.reg editing is fragile (special encoding). Try `proton run regedit` first. If that fails, the .reg import can be done on first compatdata creation via a temporary wineprefix invocation.

2. **EVIOCGRAB on source device**
   - What we know: EVIOCGRAB gives exclusive access. Prior mousegrab.go used it for mouse devices.
   - What's unclear: Should we grab the Steam Input virtual pad to prevent Wine's winebus from reading it directly? Or does DisableHidraw+Enable SDL suffice?
   - Recommendation: Do NOT grab. DisableHidraw disables winebus hidraw access. With Enable SDL, winebus reads via SDL which will see both the Steam Input pad and our virtual pad. We may need to grab the source OR use a custom VID/PID that SDL maps but the Steam Input pad's VID/PID gets filtered. Test on hardware.

3. **Hold timeout calibration**
   - What we know: ServerTravel takes >1 second. Prior debugging showed consistent zero window.
   - What's unclear: Exact duration of the zero window. Is 500ms enough? Too much?
   - Recommendation: Make configurable (default 500ms). Can be tuned on hardware. Upper bound should be ~2 seconds to avoid noticeable phantom input.

4. **Axis range scaling for Steam Input virtual pad**
   - What we know: Kernel xpad uses -32768..32767 for sticks, 0..255 for triggers. Steam Input virtual pad (28de:11ff) likely uses same ranges.
   - What's unclear: Exact ranges for Steam Input virtual pad. May differ from physical xpad.
   - Recommendation: Read actual axis ranges from source device at startup via evdev AbsoluteTypes(). Output ranges match Xbox 360 spec. Scale if source differs.
